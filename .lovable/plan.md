## Проблема

Аккаунт не создаётся из формы регистрации. По текущему состоянию проекта видно:

- В базе есть только один пользователь `admin@admin.com`; новый пользователь из формы не появляется.
- Backend доступен и работает нормально.
- Signup-форма вызывает `supabase.auth.signUp`, но при ошибке показывает только общий toast.
- В прошлых auth-логах были `422` на `/signup`, что обычно означает проблему конфигурации email/password signup, неподтверждённый/дублирующий email, слабый пароль, либо отказ auth-сервиса принять запрос.
- Триггер `handle_new_user` есть и должен создавать `profiles` + роль `student`, но он не защищён от конфликтов при повторном создании профиля/роли.

## Что исправить

1. **Починить backend-конфигурацию регистрации**
   - Проверить, что обычная email/password регистрация не отключена.
   - Оставить безопасные настройки auth без anonymous signup.
   - Не включать подтверждение email, если это не требуется явно; регистрация должна корректно объяснять пользователю следующий шаг.

2. **Сделать signup trigger идемпотентным**
   - Обновить `public.handle_new_user()` так, чтобы он не ломал создание пользователя, если профиль или роль уже существуют.
   - Для `profiles` использовать безопасный `on conflict (id) do update/do nothing`.
   - Для `user_roles` использовать `on conflict (user_id, role) do nothing`.
   - Сохранять роль `student` при создании нового аккаунта.

3. **Усилить клиентскую форму регистрации**
   - Проверять имя, email и пароль до запроса.
   - Нормализовать email.
   - Показывать точные русские ошибки для `422`, `user already registered`, `signup disabled`, `weak password`, `invalid email`, `email rate limit`.
   - После успешной регистрации:
     - если сессия создана сразу — редирект на `/dashboard`;
     - если требуется подтверждение email — редирект на `/login` с понятным сообщением.

4. **Добавить fallback-проверку после `signUp`**
   - Если `signUp` вернул пользователя без сессии, считать аккаунт созданным и не показывать ложную ошибку.
   - Если `signUp` вернул ошибку без понятного сообщения, вывести диагностируемое сообщение вместо “ничего не произошло”.

5. **Проверить результат**
   - Убедиться, что после регистрации пользователь появляется в auth users.
   - Убедиться, что автоматически создаются `profiles` и `user_roles`.
   - Проверить, что форма не остаётся в бесконечной загрузке и даёт понятный результат.

## Ожидаемый результат

После реализации:

- Кнопка `Создать аккаунт` реально создаёт пользователя.
- Пользователь получает понятное сообщение, если email уже занят, пароль слабый или регистрация временно недоступна.
- Новый аккаунт получает профиль и роль `student` автоматически.
- После регистрации пользователь попадает на правильный следующий экран.
- Dead UI больше нет: форма всегда показывает результат действия.
