## Диагностика

Проблема выглядит не как поломка кнопки, а как сбой auth-flow после нажатия «Войти»:

- Кнопка отправляет форму и вызывает вход по email/password.
- В прошлых логах backend были частые ошибки `Invalid login credentials` и `422` на signup — то есть часть аккаунтов не была создана или пароль/email не совпадали.
- В коде `login.tsx` сейчас показывается сырой текст ошибки backend, без понятного сценария восстановления.
- После успешного входа пользователь сразу уходит на `/dashboard`, где запускаются защищённые server functions. Их нужно вызывать только после гарантированной гидратации сессии, иначе возможны 401/редирект обратно на login.
- В `__root.tsx` нет глобальной инвалидации роутера/React Query при изменении auth-состояния, из-за чего UI может оставаться в старом состоянии после входа/выхода.

## Что исправить

### 1. Сделать login production-ready

В `src/routes/login.tsx`:

- Нормализовать email перед входом: `trim().toLowerCase()`.
- Добавить защиту от двойного сабмита.
- После успешного `signInWithPassword` явно дождаться актуальной сессии через `getSession()`.
- Показать понятные ошибки на русском:
  - неверный email/пароль;
  - email не подтверждён;
  - слишком много попыток;
  - общая ошибка входа.
- После успешного входа делать `toast.success` и redirect на `redirect` или `/dashboard`.

### 2. Сделать signup устойчивым

В `src/routes/signup.tsx`:

- Нормализовать email.
- Улучшить обработку `422` и случая «пользователь уже существует».
- После успешной регистрации с auto-confirm сразу отправлять в `/dashboard`.
- Если сессии нет — отправлять на `/login` с понятным сообщением.
- Не оставлять пользователя на форме без объяснения.

### 3. Исправить auth state на уровне приложения

В `src/routes/__root.tsx`:

- Подключить `supabase.auth.onAuthStateChange` один раз в root.
- При входе/выходе инвалидировать router и React Query cache.
- Не делать `await` внутри callback, чтобы не блокировать auth-события.

### 4. Укрепить protected dashboard/admin

В `src/routes/dashboard.tsx` и `src/routes/admin.tsx`:

- В `beforeLoad` учитывать `error` из `supabase.auth.getUser()`.
- На ошибку или отсутствие пользователя редиректить на `/login` с корректным `redirect`.
- В React Query для защищённых server functions отключить retry на 401/Unauthorized, чтобы не было бесконечных неудачных запросов.
- Добавить понятное состояние ошибки вместо «вечного skeleton».

### 5. Проверить backend signup trigger

Перед/во время реализации проверить, что trigger создания профиля и роли работает:

- `profiles` создаётся после signup.
- `user_roles` получает роль `student`.
- Если будет обнаружена проблема в trigger/RLS — добавить миграцию, которая чинит `handle_new_user` безопасно и идемпотентно.

## Ожидаемый результат

После исправления:

- кнопка «Войти» реально выполняет вход;
- пользователь видит понятную ошибку, если email/пароль неверные;
- новый аккаунт создаётся и попадает в кабинет;
- `/dashboard` не выбрасывает обратно из-за гонки сессии;
- logout/login обновляют состояние сайта без перезагрузочных артефактов;
- auth-flow готов к production.
